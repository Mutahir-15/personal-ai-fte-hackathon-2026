# Implementation Plan: Silver Tier Communication & Social Automation

**Branch**: `002-silver-tier-communication` | **Date**: 2026-03-14 | **Spec**: [specs/002-silver-tier-communication/spec.md](spec.md)
**Input**: Feature specification from `/specs/002-silver-tier-communication/spec.md`

## Summary

The Silver Tier extends the Personal AI Employee with a robust communication layer. This includes Gmail monitoring via OAuth2, WhatsApp message detection via Playwright, LinkedIn post generation and publishing via Gemini CLI, and a local Node.js MCP server for email actions. A core component is the Silver HITL Approval handler, which manages priority-ordered approval queues and escalation alerts in the Obsidian vault.

## Technical Context

**Language/Version**: Python 3.13+, Node.js v24+ LTS (for MCP)
**Primary Dependencies**: `google-auth-oauthlib`, `google-api-python-client`, `playwright`, `requests`, `requests-oauthlib`, `@modelcontextprotocol/sdk`, `googleapis`
**Storage**: Local files (JSON/Markdown) in Obsidian Vault (`VAULT_PATH`)
**Testing**: `pytest` for logic, manual verification for OAuth2/QR flows
**Target Platform**: Windows 10
**Project Type**: Mixed Python (Services/Watchers) and Node.js (MCP Server)
**Performance Goals**: 120s polling for Gmail, 30s for WhatsApp, 60s UI refresh
**Constraints**: `DRY_RUN` default, Windows paths (`\\`), `msvcrt` file locking
**Scale/Scope**: Communication automation for a single professional user

## Implementation Strategy

### PHASE 1 — Security Foundation
- Extend `.env` and `.gitignore` for Silver secrets.
- Verification: Confirm secrets are ignored and `DRY_RUN=true`.

### PHASE 2 — Gmail Watcher
- Implement `gmail_watcher.py` using `google-auth-oauthlib`.
- Drops `.md` files into `/Needs_Action/`.
- Handle OAuth2 first-run manually before PM2 registration.

### PHASE 3 — WhatsApp Watcher
- Implement `whatsapp_watcher.py` using Playwright.
- Persistent browser context for session storage.
- QR scan required on first run (headless: false).

### PHASE 4 — Email MCP Server
- Node.js server exposing `send_email`, `draft_email`, `search_emails`, `get_email`, `reply_email`.
- Registered in Gemini CLI `mcp.json`.
- Enforcement: `send_email` requires file in `/Approved/`.

### PHASE 5 — LinkedIn Poster
- `linkedin_poster.py` using Gemini CLI for generation.
- Publishing via LinkedIn API v2 `ugcPosts`.
- Mandatory HITL approval before publishing.

### PHASE 6 — Silver HITL Approval
- `silver_hitl_approval.py` replaces Bronze `hitl_approval.py`.
- Priority-ordered queue and escalation alerts (4h/8h/20h).
- Backward compatibility with Bronze types.

### PHASE 7 — Orchestrator & PM2 Extension
- Extend `orchestrator.py` with Silver event handlers.
- Register all new services in `ecosystem.config.js`.

## Security Plan
- **Secret Management**: All tokens (`gmail_token.json`, `linkedin_token.json`) and session folders (`whatsapp_session/`) added to `.gitignore`.
- **Environment**: All API keys and Client IDs stored in `.env` only.
- **Enforcement**: `DRY_RUN=true` by default for all new scripts.

## Error Recovery
- **Watchers**: PM2 auto-restart with state restoration from JSON files.
- **Sessions**: Expired sessions trigger Dashboard alerts for manual intervention (QR scan/OAuth re-auth).
- **MCP**: Orchestrator detects offline status and queues actions.
